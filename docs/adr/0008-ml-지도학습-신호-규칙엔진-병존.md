# ADR 0008: ML 지도학습 신호를 규칙 기반 추천과 병존

**상태**: 채택됨

## 맥락

기존 추천은 재무·차트·뉴스 점수를 사람이 정한 고정 가중치(예: 40/40/20)로 더하는
**규칙 기반**(`src/analyzers/recommendation_engine.py`)이다. 이 가중치가 실제로
수익이 되는 판단인지 **과거 데이터로 검증된 적이 없고**, "왜 이 등급인가"가 직관에
의존한다.

`/grill-me` 세션에서, "과거에 이런 상황에서 샀다면 비용 빼고 돈을 벌었는가"를 데이터로
학습하는 **지도학습 신호**를 도입하기로 했다. 문제는 이것을 규칙 엔진과 어떤 관계로
둘 것인가다: 대체할 것인가, 병존시킬 것인가.

## 결정

지도학습 신호(`src/ml/`, LightGBM)를 **규칙 엔진과 병존**시킨다. 규칙 엔진을 **대체하지
않는다.**

- 규칙 엔진 코드(`recommendation_engine.py`)는 이번 작업에서 **변경하지 않는다.**
- ML 신호는 독립 패키지 `src/ml/`로 격리한다(수집→라벨→피처→분할→모델→백테스트).
- 두 시스템 중 무엇을 쓸지(또는 ML로 규칙 엔진을 대체할지)는 ML 신호가 **성공 판정을
  통과한 뒤**(백테스트에서 두 기준선을 꾸준히 이긴 뒤) 별도로 결정한다.
- 모델은 **LightGBM/XGBoost**(트리)로 한다. 표(tabular) 기술지표 + 해석가능성 요건 +
  제한적 유효 표본에서 트리가 적합하고, 매 예측당 TreeSHAP 근거를 낼 수 있다.
  트랜스포머는 이 규모·해석 요건에서 부적합 → 범위 밖.

## 결과

**긍정**
- 규칙 엔진의 안정성을 해치지 않고 학습 기반 접근을 실험할 수 있다(리스크 격리).
- 라벨=돈·미래참조 통제·walk-forward 성공 판정으로, 규칙 엔진에 없던 **검증 가능성**을
  확보한다.
- TreeSHAP으로 개별 신호의 근거를 제시할 수 있다(단, 상관이지 인과 아님).

**부정 / 트레이드오프**
- 두 추천 체계가 병존해 **사용자에게 어느 것을 신뢰할지 혼란**을 줄 수 있다. v1에서는
  ML 신호를 앱에 노출하지 않아(별도 문서 참고) 이 혼란을 회피한다.
- **v1 실측상 ML 신호는 대형주에서 엣지가 없다.** 규칙 엔진을 대체할 근거가 아직 없고,
  ML 도입의 가치는 v2(재무·뉴스 추가 등) 이후에 재평가해야 한다.
- 유지보수 대상이 둘로 늘어난다.
