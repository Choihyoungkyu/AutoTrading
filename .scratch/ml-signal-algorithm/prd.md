# ML 기반 매수 신호 알고리즘 (v1: 차트 전용)

> 이 PRD는 `/grill-me` 세션에서 도출된 스펙을 기반으로 한다. 기존 규칙 기반 추천 엔진(`recommendation_engine.py`, 재무·차트·뉴스 가중평균)과는 **별개의 신규 알고리즘 모듈**이다. 기존 엔진은 이 PRD에서 수정하지 않는다.

## Problem Statement

직장인 개인 투자자는 "이 종목을 지금 사도 될까?"를 판단할 근거가 없다. 현재 시스템은 재무·차트·뉴스 점수를 사람이 정한 고정 가중치(30/40/30)로 더한 **규칙 기반** 추천만 제공한다. 이 가중치가 실제로 돈이 되는 판단인지 **과거 데이터로 검증된 적이 없고**, "왜 이 등급인지"가 사람의 직관에 의존한다.

투자자 관점에서 진짜 원하는 것은: **"과거에 실제로 이런 상황에서 샀다면 (비용 빼고) 돈을 벌었는가?"**를 데이터로 학습한 신호, 그리고 그 신호가 **미래참조(look-ahead) 없이** 정직하게 검증되었다는 신뢰다.

## Solution

투자자 관점에서, 시스템은 각 종목·각 시점에 대해 **"매수 / 매수 안 함"** 이진 신호와 그 **신뢰도(확률)**, 그리고 **그 판단의 근거(어떤 기술지표가 신호를 밀었는지)**를 제시한다.

이 신호는 지도학습 모델(LightGBM)이 만든다. 학습의 정답(라벨)은 "그 시점에 샀다면 손절 -3% 또는 5거래일 뒤 청산까지 실제로 돌렸을 때 **거래비용을 뺀 실현손익이 (+)였는가**"이다. 즉 **라벨 = 실제로 번 돈**이다.

가장 중요한 약속은 정직성이다: 모든 입력 피처는 **결정 시점에 실제로 존재했던 데이터만**으로 계산되고(인과적), 학습/검증은 시간 순서를 지키며(walk-forward + embargo), 최종 성능은 **바보 전략과 코스피 지수를 비용 차감 후에도 이겼는지**로 판정한다.

**v1 범위:** 입력은 **차트/기술지표만** 사용한다. 재무·뉴스는 point-in-time 구축 난이도(재무 공시 시차, 뉴스 timestamp) 때문에 v2로 미룬다.

## User Stories

1. 개인 투자자로서, 나는 종목코드를 주면 "매수 / 매수 안 함" 신호를 받고 싶다. 그래야 지금 진입해도 될지 판단할 수 있다.
2. 개인 투자자로서, 나는 신호와 함께 **매수 확률(0~1)**을 보고 싶다. 그래야 얼마나 확신 있는 신호인지 알 수 있다.
3. 개인 투자자로서, 나는 매수 신호마다 **근거(어떤 기술지표가 이 판단을 밀었는지)**를 보고 싶다. 그래야 블랙박스를 맹신하지 않고 스스로 납득할 수 있다.
4. 개인 투자자로서, 나는 신호가 **미래참조 없이** 검증됐음을 알고 싶다. 그래야 백테스트 숫자를 신뢰할 수 있다.
5. 개인 투자자로서, 나는 이 신호를 따라 매매했을 때의 **비용 제외 기대수익**을 알고 싶다. 그래야 수수료·세금에 파묻히는 신호인지 판단할 수 있다.
6. 개인 투자자로서, 나는 이 모델이 **"그냥 다 사는 전략"과 "코스피 지수 보유"보다 나은지** 비교 결과를 보고 싶다. 그래야 이 모델을 쓸 이유가 있는지 안다.
7. 데이터 담당자로서, 나는 각 시점의 **유동성 상위 ~200 종목 유니버스**를 재현 가능하게 뽑고 싶다. 그래야 소형주 슬리피지와 상폐 리스크를 줄인 상태로 학습할 수 있다.
8. 데이터 담당자로서, 나는 종목의 일봉 OHLCV를 표준 DataFrame 형태로 얻고 싶다. 그래야 이후 라벨·피처 단계가 수집원(PYKRX)에 의존하지 않는다.
9. 알고리즘 담당자로서, 나는 OHLCV 시계열과 파라미터(손절률·기간·비용)를 주면 **0/1 라벨**을 얻고 싶다. 그래야 "실제로 번 돈"을 정답으로 학습할 수 있다.
10. 알고리즘 담당자로서, 나는 라벨 생성기가 **손절 발동 경로**(중간에 -3% 터치 시 그날 청산)를 정확히 반영하길 원한다. 그래야 라벨과 실제 손익이 일치한다.
11. 알고리즘 담당자로서, 나는 라벨 임계치가 단순 0%가 아니라 **왕복 거래비용을 초과한 (+)**이길 원한다. 그래야 비용에 묻히는 미미한 수익을 "매수 정답"으로 오라벨하지 않는다.
12. 알고리즘 담당자로서, 나는 시점 T까지의 데이터만으로 **인과적 기술지표**를 계산하고 싶다. 그래야 이동평균·RSI 등에 미래 봉이 새지 않는다.
13. 알고리즘 담당자로서, 나는 피처 빌더에 T 이후 봉을 추가로 넣어도 **T 시점 피처가 변하지 않음**을 보장받고 싶다. 그래야 미래참조가 없음을 코드로 증명할 수 있다.
14. 알고리즘 담당자로서, 나는 학습/검증을 **시간 순서로만** 나누고 싶다. 그래야 미래로 과거를 맞히는 누수를 막는다.
15. 알고리즘 담당자로서, 나는 학습셋과 검증셋 사이에 **최소 5거래일 embargo**를 두고 싶다. 그래야 겹치는 창(overlapping window)으로 인한 라벨 누수를 막는다.
16. 알고리즘 담당자로서, 나는 여러 시기로 굴려가는 **walk-forward 검증**을 하고 싶다. 그래야 성능이 운인지 실력인지 구간별 일관성으로 판단한다.
17. 알고리즘 담당자로서, 나는 마지막 한 구간을 **금고(lockbox)로 봉인**하고 싶다. 그래야 튜닝 과정에서 서서히 스며드는 미래참조 없이 최종 점수를 딱 한 번 확인한다.
18. 알고리즘 담당자로서, 나는 표(tabular) 기술지표에 **LightGBM/XGBoost**를 쓰고 싶다. 그래야 이 데이터 규모에서 과적합을 피하고 해석 가능성을 확보한다.
19. 알고리즘 담당자로서, 나는 매 예측마다 **TreeSHAP 값**을 뽑고 싶다. 그래야 개별 신호의 근거(스토리 3번)를 제공할 수 있다.
20. 평가 담당자로서, 나는 신호를 확률순으로 정렬해 **상위 신호만 매매하는 백테스트**를 하고 싶다. 그래야 전체 정확도가 아니라 실제로 거래할 신호의 수익을 본다.
21. 평가 담당자로서, 나는 백테스트 수익을 **바보 전략(다 사서 5일 뒤 청산)**과 **코스피 지수 보유** 두 기준선과 비교하고 싶다. 그래야 모델의 초과 성과를 판정한다.
22. 평가 담당자로서, 나는 초과 성과가 **왕복 거래비용의 2배 이상 마진**으로 **walk-forward 구간 대다수에서** 나오는지 보고 싶다. 그래야 성공/실패를 명확한 기준으로 선언한다.
23. 평가 담당자로서, 나는 결과 보고에 **생존 편향으로 성능이 낙관적으로 나온다**는 한계가 명시되길 원한다. 그래야 숫자를 과신하지 않는다.
24. 유지보수자로서, 나는 v1이 차트만 쓴다는 점과 재무·뉴스가 v2 과제임이 문서에 남길 원한다. 그래야 범위 혼동이 없다.

## Implementation Decisions

### 파이프라인 개요 (신규 모듈)

`src/`에 ML 신호 파이프라인을 신규로 추가한다. 기존 규칙 기반 `recommendation_engine.py`는 건드리지 않는다. 파이프라인은 다음 단계로 구성되며, **핵심 입력 계약은 "일봉 OHLCV DataFrame"**이다 — 이 계약 덕분에 라벨·피처·분할·백테스트가 수집원(PYKRX)에 의존하지 않는다.

1. **유니버스 선정** — 각 결정 시점 기준 거래대금/시가총액 상위 ~200 종목을 그 시점 데이터로 재선정(point-in-time 근사). 기존 `krx_collector.py`(PYKRX) 위에 얹는다.
2. **데이터 수집** — 종목별 일봉 OHLCV를 표준 DataFrame으로 반환. 수정주가 사용 시 발생할 수 있는 미래 소급 정보 유입에 주의(원본 가격 + 별도 조정계수 접근을 우선 검토).
3. **라벨 생성기 (S1)** — 실현손익 시뮬레이터.
4. **인과적 피처 빌더 (S2)** — 기술지표.
5. **Walk-forward 분할기 (S3)** — embargo + lockbox.
6. **모델 학습** — LightGBM/XGBoost.
7. **백테스트 평가기 (S4)** — 기준선 대비 성공 판정 + TreeSHAP.

### 결정 사항

- **문제 유형:** 이진 분류 (`1` = 매수, `0` = 매수 안 함). 예측 기간 **5거래일(1주)** 단일. (그릴링 초기의 1일/1개월 지평은 v1 범위에서 제외.)

- **라벨 정의 (S1, 파이프라인의 심장):** 시점 T에 매수했다고 가정하고 다음 매매 규칙을 그대로 시뮬레이션한 **실현손익**으로 라벨을 만든다.
  - 진입: T 종가 매수 (또는 명시된 진입가 규칙).
  - 청산 규칙: T+1 ~ T+5 중 **장중 저가가 -3% 손절선을 터치하면 그 시점 손절가로 청산**, 아니면 **T+5 종가로 청산**.
  - 라벨: `실현손익(%) - 왕복 거래비용(%) > 0` 이면 `1`, 아니면 `0`. 비용은 수수료+거래세+슬리피지로 대략 0.5~1% 파라미터화.
  - 라벨은 "T+5 종가만 보는" 방식(look-ahead의 (A))이 **아니라** 위 경로 의존적 실현손익이다. (그릴링에서 라벨=돈으로 확정.)

- **입력 피처 (S2):** **차트/기술지표만.** 시점 T까지의 봉만으로 인과적으로 계산(이동평균, RSI, 거래량 파생, 변동성 등). 전체 기간 기준 정규화 금지(미래참조). 지표 목록은 구현 시 확정하되, 각 지표는 "T 이후 데이터에 불변" 계약을 지킨다.

- **유니버스:** 각 시점 유동성 상위 ~200 종목(코스피 대형주 중심). 소형주 제외로 슬리피지·상폐 리스크 완화. 생존 편향은 v1에서 완전 제거하지 않고 **한계로 명시**(상폐 종목 포함 데이터는 v2).

- **분할 (S3):** 시간 순서 고정 walk-forward. 학습–검증 사이 **5거래일 embargo**. 마지막 구간은 **lockbox**로 봉인해 최종 1회만 개봉. 예시 구성(데이터 가용 범위에 맞춰 조정):
  - `학습 2018–2021 → 검증 2022`
  - `학습 2018–2022 → 검증 2023`
  - `학습 2018–2023 → 검증 2024 (lockbox)`

- **모델:** LightGBM 또는 XGBoost. 트랜스포머는 v1 오답(표 데이터 부적합, 이 규모에서 과적합, 해석 불가) — v3 이후 재검토.

- **해석 (스토리 3):** 매 예측당 **TreeSHAP** 기여도 상위 피처를 근거로 제시. 단, SHAP는 상관이지 인과가 아님을 산출물에 표기.

- **평가/성공 지표 (S4):** 신호를 확률 내림차순 정렬 → **상위 신호만 매매**하는 백테스트. 성과 = **비용 제외 누적수익**. 비교 기준선 2개: ① 바보 전략(다 사서 5일 뒤 청산), ② 코스피 지수 보유.
  - **성공 정의:** walk-forward 구간 **대다수(예: 3개 중 2개 이상)**에서, 상위 신호 매매의 비용 제외 수익이 두 기준선을 **왕복 거래비용의 2배 이상 마진**으로 상회. 한 구간만 좋으면 운으로 간주해 실패.

- **모듈 인터페이스(개념):** 각 단계는 순수 함수/결정적 컴포넌트로 노출한다.
  - 라벨 생성기: `(OHLCV, {손절률, 기간, 비용}) -> 라벨 시리즈(0/1)`
  - 피처 빌더: `(OHLCV, 시점 T) -> 피처 벡터` (T 이후 불변)
  - 분할기: `(날짜 범위, {embargo, 폴드 정의}) -> [(학습 구간, 검증 구간), ...]`
  - 백테스트 평가기: `(신호+확률, 가격, {비용, top-k}) -> {누적수익, 기준선 대비, 구간별 성과}`

- **API 노출:** v1에서 신규 알고리즘의 API 노출은 최소화(선택). 노출 시 기존 `routes.py` 관례를 따르되, 규칙 기반 추천 엔드포인트와 **경로를 분리**해 혼동을 막는다.

## Testing Decisions

**좋은 테스트의 기준:** 구현 세부가 아니라 **외부로 관찰 가능한 동작**만 검증한다. 즉 특정 지표 계산의 내부 공식이 아니라, "이 입력 시계열에 대해 라벨이 이렇게 나온다 / T 이후 봉을 넣어도 피처가 안 바뀐다 / 분할에 embargo 갭이 존재한다 / 상위 신호 수익이 기준선 대비 이렇게 계산된다"를 확인한다.

**선행 사례(prior art):** 기존 `tests/test_recommendation.py`, `test_financial.py`, `test_price_target.py`는 **고정 입력 단위 테스트**(예: 고정 점수로 buy/hold/sell 경계 검증) 방식을 쓴다. 신규 테스트도 동일하게 **합성/고정 OHLCV DataFrame**을 입력으로 결정적으로 검증한다. **PYKRX는 목킹하지 않는다** — 4개 seam 모두 DataFrame만 받으므로 수집원 없이 테스트된다.

**테스트할 모듈(seam):**

- **S1 라벨 생성기** — 최우선. 합성 가격 시나리오로 전 경로 검증:
  - 5일 내내 상승만 → 손절 미발동, T+5 종가 청산, 비용 초과 수익 → 라벨 `1`.
  - 2일차 장중 -3% 터치 후 반등 → 손절 발동해 -3% 청산 → 라벨 `0` (T+5 종가가 +3%여도 라벨은 0이어야 함 — 라벨=돈 검증).
  - 소폭 상승(+0.4%)이 비용에 못 미침 → 라벨 `0` (비용 초과 임계치 검증).
  - 경계값: 정확히 손절선/비용선에 걸치는 경우의 처리.
- **S2 인과적 피처 빌더** — 미래참조 부재 검증:
  - 시점 T의 피처를 계산 → 이후 데이터(T+1..)를 추가로 붙여 재계산 → **T 시점 피처가 동일**해야 함.
  - 지표 워밍업 기간(초기 구간)에 대한 결측/NaN 처리.
- **S3 Walk-forward 분할기** — 결정적 구조 검증:
  - 검증 구간이 학습 구간보다 **항상 미래**에 있음.
  - 학습 끝과 검증 시작 사이 **정확히 ≥5거래일 갭**.
  - lockbox 구간이 반환 폴드에 **포함되지 않거나 별도 표시**됨.
- **S4 백테스트 평가기** — 손익/비교 로직 검증:
  - 고정 신호+가격으로 상위 top-k 매매 수익이 **비용 차감 후** 정확히 계산됨.
  - 바보 전략·지수 기준선 수익 계산의 정확성.
  - 성공 판정 로직(구간 대다수 + 2배 마진)이 경계 케이스에서 옳게 True/False.

**테스트하지 않는 것:** LightGBM/XGBoost 라이브러리 내부 학습 동작(외부 라이브러리), PYKRX 네트워크 호출(수집 계층은 별도, seam 위에서 DataFrame으로 대체).

## Out of Scope

- **재무·뉴스 피처.** v2로 미룸. 재무는 공시 시차(분기말+45일, 연간+90일) 강제 근사 필요, 뉴스는 발행 timestamp 필터와 감성모델 누수 검토 필요.
- **생존 편향 완전 제거.** v1은 대형주 한정 + 한계 명시로 완화만. 상폐/거래정지 종목 포함 데이터셋은 v2.
- **1일/1개월 예측 지평.** v1은 5거래일 단일.
- **트랜스포머 계열 모델.** 데이터 규모/해석 요건상 v1 부적합. v3 이후 재검토.
- **실시간·라이브 매매 연동, 포지션 사이징, 다종목 포트폴리오 최적화.** v1은 단일 신호 생성·백테스트까지.
- **기존 규칙 기반 추천 엔진 교체.** 신규 모듈은 병존하며, 규칙 엔진 대체 여부는 성공 판정 후 별도 결정.

## Further Notes

- 이 프로젝트의 성패는 모델 선택(GBM vs 트랜스포머)이 아니라 **라벨·데이터의 미래참조 통제(S1·S2·S3)**에서 갈린다. 모델은 마지막 10%다.
- 구현 순서 권장: **S1 라벨 생성기 → S2 피처 빌더 → S3 분할기 → 수집/유니버스 → 모델 학습 → S4 백테스트 → lockbox 1회 개봉.**
- SHAP 근거는 "모델의 자기 변명"이지 "시장의 이유"가 아니다. 산출물에 이 한계를 함께 표기할 것.
